# DOCX 편집 핵심 원칙

## 최우선 원칙: 서식 보존

DOCX 편집에서 가장 중요한 것은 **기존 서식을 깨뜨리지 않는 것**이다.

숫자 하나 잘못 넣는 것보다 서식이 깨지는 것이 더 큰 문제다.
서식이 깨지면 전체 문서를 다시 작업해야 할 수 있다.

서식이 깨질 위험이 있으면 **차라리 변경하지 않는다**.

## Clone & Modify 원칙

**새 XML 요소를 절대 직접 생성하지 마라.**

반드시 기존 요소를 deepcopy(완전 복제)한 후 내용만 수정한다.

| 작업 | 올바른 방법 | 잘못된 방법 |
|------|-----------|-----------|
| 셀 값 변경 | 기존 Run의 텍스트만 교체 | Run을 삭제하고 새로 생성 |
| 행 추가 | 인접 행을 deepcopy 후 텍스트 교체 | 빈 행을 새로 만듬 |
| 테이블 추가 | 인접 테이블을 deepcopy 후 내용 교체 | 테이블을 처음부터 생성 |
| 문단 추가 | 인접 문단을 deepcopy 후 텍스트 교체 | 빈 문단을 새로 만듬 |

### 왜 Clone & Modify인가?

DOCX의 서식 정보는 XML 속성에 숨어있다:
- `<w:rPr>`: Run 속성 (폰트, 크기, 볼드, 이탤릭, 색상...)
- `<w:pPr>`: 문단 속성 (정렬, 줄간격, 들여쓰기...)
- `<w:tcPr>`: 셀 속성 (테두리, 배경색, 수직정렬...)
- `<w:trPr>`: 행 속성 (높이, 페이지 나누기...)
- `<w:tblPr>`: 테이블 속성 (테두리, 너비, 정렬...)

이 속성들을 하나라도 빠뜨리면 서식이 깨진다.
deepcopy하면 이 모든 속성이 그대로 복제된다.

## 변경 순서

1. **삭제 먼저, 높은 인덱스부터** (인덱스 밀림 방지)
2. **수정** (기존 요소 텍스트만 교체)
3. **추가 마지막** (새 요소는 clone 후 삽입)

```
예시: body에 요소 [0, 1, 2, 3, 4]가 있을 때
- 요소 3 삭제 + 요소 1 수정 + 요소 2 뒤에 추가

올바른 순서:
  1. 요소 3 삭제 → [0, 1, 2, 4]
  2. 요소 1 수정 → [0, 1', 2, 4]
  3. 요소 2 뒤에 추가 → [0, 1', 2, NEW, 4]

잘못된 순서 (추가 먼저):
  1. 요소 2 뒤에 추가 → [0, 1, 2, NEW, 3, 4]
  2. 요소 3 삭제 → NEW가 삭제됨! (인덱스가 밀렸으므로)
```

## 한 번에 하나씩

대량 변경 대신, 한 테이블, 한 행, 한 셀씩 변경하고 매번 확인.

```
좋은 패턴:
  set_cell_text(table=3, row=5, col=2, text="1,234,567")
  read_cell(table=3, row=5, col=2)  → "1,234,567" ✓
  set_cell_text(table=3, row=6, col=2, text="456,789")
  read_cell(table=3, row=6, col=2)  → "456,789" ✓

나쁜 패턴:
  (50개 셀을 한 번에 변경하고 마지막에 한 번만 확인)
  → 어디서 잘못됐는지 알 수 없음
```

## 검색 반경

body에서 특정 인덱스의 요소를 찾을 때, 해당 인덱스에 없으면 ±3 범위 내 검색.
(삭제/추가로 인덱스가 밀렸을 수 있으므로)

## 절대 하지 말 것

1. XML을 직접 문자열로 조립하지 마라
2. `<w:r>` (Run)을 새로 만들지 마라 (기존 것의 rPr을 deepcopy)
3. 스타일 속성을 추측하지 마라 (반드시 기존 것에서 복사)
4. 테이블 구조(열 수, gridCol)를 변경하지 마라
5. 병합(vMerge, gridSpan)을 임의로 수정하지 마라

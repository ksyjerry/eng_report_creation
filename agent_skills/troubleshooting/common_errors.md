# 자주 발생하는 오류와 해결법

## 1. 값이 엉뚱한 열에 들어감

**증상**: 숫자가 한 칸 옆에 기록되거나, spacer column에 들어감

**원인**: DOCX에 spacer column이 있어 logical index와 physical index가 다름

**진단**:
- `get_column_info(table)` 실행
- spacer column 존재 여부 확인
- 데이터가 들어간 physical column이 spacer인지 확인

**해결**: physical column index로 변환하여 사용
- 참조: editing/column_mapping.md

---

## 2. 연도가 전부 같은 값으로 변환됨

**증상**: "2024"와 "2023"이 전부 "2025"로 됨

**원인**: 순차적으로 교체할 때 cascade 발생
- "2023"→"2024" 실행 후
- "2024"→"2025" 실행 시 방금 바뀐 것도 잡힘

**진단**: 문서에서 "2024"가 있어야 할 곳에 "2025"가 있으면 cascade 의심

**해결**: 모든 연도 교체를 한 번에 atomic하게 수행
- `replace_in_headers_footers([("2024","2025"), ("2023","2024")])` 사용
- 참조: editing/year_rolling.md

---

## 3. 연도가 일부 변환 안 됨

**증상**: header/footer 또는 테이블 헤더에 이전 연도가 남아있음

**원인**: Word가 텍스트를 여러 Run으로 분할 저장
- "2024"가 "202" + "4" 또는 "20" + "24"로 분할
- 단순 텍스트 검색으로 찾을 수 없음

**진단**:
- `read_header_footer()` 결과에 이전 연도가 보이면
- 해당 XML을 직접 확인하여 Run 분할 여부 확인

**해결**: cross-run 교체 사용 (문단 전체 텍스트를 연결 → 교체 → 재분배)
- 참조: editing/year_rolling.md

---

## 4. 행 매칭이 엉뚱하게 됨

**증상**: DSD의 값이 DOCX의 다른 항목 행에 들어감
- 예: "매출원가" 값이 "판매비" 행에 들어감

**원인**: 라벨 유사도만으로 매칭하면 비슷한 다른 행과 혼동
- "감가상각비"와 "무형자산상각비"가 둘 다 "Depreciation"에 매칭
- "평가이익"과 "평가손실"이 유사도가 높음

**진단**: 매칭된 행의 DSD 전기값과 DOCX 당기값을 비교
- 값이 크게 다르면 (비율 > 2.0) 잘못된 매칭

**해결**: 라벨 유사도 + 값 검증을 함께 사용
- DSD 전기값 == DOCX 당기값이면 올바른 매칭
- 참조: matching/value_validation.md

---

## 5. 셀 서식이 깨짐

**증상**: 특정 셀의 폰트, 크기, 정렬, 테두리가 변경됨

**원인**: Run을 새로 생성하거나, 기존 Run의 속성(rPr)을 덮어씀

**진단**: 변경 전후 셀의 시각적 비교

**해결**: 반드시 기존 Run을 보존하고 텍스트만 교체
- `set_cell_text()`가 rPr을 deepcopy하는지 확인
- 참조: editing/cell_editing.md

---

## 6. 빈 셀이 숫자로 인식됨

**증상**: spacer column이나 빈 셀이 데이터 열로 감지됨

**원인**: 빈 문자열("")이나 대시("-")가 숫자로 판정됨

**진단**: 기간 열 감지 결과가 spacer column을 가리키면 의심

**해결**: 실제 숫자값(콤마 포함 숫자)이 있는 셀만 데이터 열로 판정
- 빈 셀, 대시, 백슬래시는 숫자에서 제외

---

## 7. 위치 매칭으로 엉뚱한 행 연결

**증상**: 테이블 구조가 다른데 같은 위치끼리 매칭되어 엉뚱한 값 입력

**원인**: 두 테이블의 행 수만 같으면 위치 매칭을 시도
- 하지만 행의 의미가 완전히 다를 수 있음
- 예: DSD "회사명" 행이 DOCX "Location" 행에 매칭

**진단**: 위치 매칭된 행의 숫자 셀이 전혀 겹치지 않으면 의심

**해결**: 위치 매칭 후 반드시 숫자값 overlap 검증
- 겹치는 숫자가 없으면 매칭 거부
- 참조: matching/value_validation.md

---

## 8. vMerge 행의 값 누락

**증상**: 수직 병합된 셀에서 첫 행만 업데이트되고 나머지는 빈칸

**원인**: vMerge continuation 셀은 rowspan=0으로 표시되어 건너뛰어짐

**진단**: 수직으로 병합된 영역의 셀이 비어있으면 의심

**해결**: vMerge restart 셀(병합 시작점)에만 값을 넣으면 병합 영역 전체에 반영됨
- continuation 셀은 건드리지 않아도 됨

---

## 9. 테이블 인덱스가 안 맞음

**증상**: 특정 인덱스의 테이블이 예상과 다른 테이블

**원인**: 삭제/추가 작업으로 body 인덱스가 밀림

**진단**: `read_table()` 결과가 예상과 다르면 인덱스 밀림 의심

**해결**:
- 높은 인덱스부터 삭제
- 삭제 후 인덱스 재계산
- ±3 범위 검색 활용
- 참조: editing/docx_editing.md

---

## 10. 단위 불일치

**증상**: 모든 숫자가 1,000배 또는 1/1,000배 차이남

**원인**: DSD는 "원" 단위, DOCX는 "천원" 단위 (또는 반대)

**진단**: 모든 숫자가 동일 비율로 차이나면 단위 문제

**해결**: 테이블 상단의 단위 정보 확인
- DSD: "(단위: 원)" vs "(단위: 천원)"
- DOCX: "(In Korean won)" vs "(In thousands of Korean won)"
- 단위 변환 후 적용

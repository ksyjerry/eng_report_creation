# 테이블 행 매칭

## 개요
DSD 테이블의 행과 DOCX 테이블의 행을 매칭하는 작업.
**한국어 라벨 vs 영어 라벨**이므로 단순 문자열 비교는 불가능하다.

## 매칭 전략 (우선순위 순)

### 전략 1: 값 기반 매칭 (가장 신뢰도 높음)

**원리**: DSD 전기값 == DOCX 당기값이면 같은 행이다.

전기 DOCX의 당기 열에 있는 숫자는 = DSD 전기 열의 숫자.
따라서 **DSD 전기값과 DOCX 당기값이 같으면** 같은 항목이라고 판단할 수 있다.

```
DSD:   유동자산  |  1,500,000 (당기)  |  1,200,000 (전기)
DOCX:  Current assets  |  1,200,000 (당기=전기 DOCX)  |  900,000 (전기)

→ DSD 전기(1,200,000) == DOCX 당기(1,200,000) → 매칭!
```

**주의사항**:
- 같은 값을 가진 행이 여러 개 있을 수 있음 → 라벨과 함께 검증
- 0이나 대시(-)는 매칭에 사용하지 않음 (너무 많은 행이 해당)
- 음수는 절대값으로 비교 (괄호 표현 차이 있을 수 있음)

### 전략 2: 라벨 번역 후 매칭

**원리**: 한국어 라벨을 영어로 번역한 후 DOCX 라벨과 비교.

번역 소스 (우선순위):
1. **Glossary** (같은 문서 쌍에서 구축): 가장 정확
2. **IFRS 표준 용어** (ifrs_terms.md 참조): 표준적인 항목
3. **LLM 번역**: 나머지

```
DSD 라벨: "유동자산"
→ IFRS 사전: "Current assets"
→ DOCX에서 "Current assets" 검색 → 매칭!
```

**유사도 판단**:
- 정확히 일치: 신뢰도 1.0
- 한쪽이 다른 쪽을 포함: 신뢰도 0.7~1.0 (길이 비율에 따라)
- 단어 겹침(Jaccard): 겹치는 단어 수 / 전체 단어 수 × 0.9
- 신뢰도 0.6 이상이면 매칭 후보

### 전략 3: 위치 기반 매칭 (최후 수단)

**원리**: 같은 위치의 행은 같은 항목일 가능성이 높다.

**사용 조건**:
- 전략 1, 2로 매칭 안 된 행이 남아있을 때
- 두 테이블의 데이터 행 수가 같을 때만
- 반드시 값 검증을 함께 수행

**값 검증**:
- 위치로 매칭한 후, 해당 행들의 숫자 셀이 겹치는지 확인
- 겹치면 신뢰도 0.65, 안 겹치면 0.35 (→ 0.5 미만이므로 변경 생성 안 됨)

## 라벨 추출

행의 첫 번째 비숫자 셀의 텍스트가 라벨이다.

```
| 유동자산 | 1,234,567 | 1,123,456 |
  ^^^^^^^^
  이것이 라벨
```

**정규화**:
- 앞뒤 공백 제거
- 연속 공백을 하나로
- 소문자 변환 (영어)
- 선행 번호 제거: "1) 현금" → "현금", "(1) 매출" → "매출"

## 숫자 판별

다음은 "숫자"로 판별:
- `1,234,567` — 천단위 콤마 포함 숫자
- `(1,234,567)` — 괄호 음수
- `-1,234,567` — 마이너스 음수
- `-` / `—` / `–` — 대시 (0으로 취급)

다음은 "숫자 아님":
- 빈 문자열 `""`
- 공백만 `"   "`
- 백슬래시 `"\"`
- 텍스트 (한글, 영어 등)

**중요**: 빈 셀("")과 대시("-")는 숫자 컬럼 감지에서 제외해야 한다!
빈 셀을 숫자로 취급하면 spacer column이 데이터 열로 잘못 감지됨.

## 기간(Period) 열 감지

DSD 헤더의 기간 마커:
- 당기: "당기", "당기말", "당분기", "당분기말"
- 전기: "전기", "전기말", "전분기", "전분기말"

DOCX 헤더의 기간 마커:
- 당기: "2025", "December 31, 2025"
- 전기: "2024", "December 31, 2024"

**열 검증**: 감지된 기간 열에 실제 숫자 데이터가 있는지 확인.
헤더의 gridSpan이 spacer column을 포함할 수 있으므로,
감지된 열이 spacer이면 인접 열로 보정.

## 흔한 실수

1. **라벨만으로 매칭 → 엉뚱한 행 연결**
   - "감가상각비"가 "Depreciation"과 "Amortization" 두 곳에 비슷하게 매칭
   - → 반드시 값 검증으로 교차확인

2. **당기/전기 열 혼동**
   - DSD의 당기 ↔ DOCX의 전기를 매핑해야 하는데 당기↔당기로 매핑
   - → DSD 전기값 == DOCX 당기값이 올바른 검증

3. **헤더 행 수 차이**
   - DSD는 헤더 1행, DOCX는 헤더 2행이면 데이터 행 인덱스가 1 차이남
   - → 헤더 행을 제외한 데이터 행끼리만 매칭

4. **빈 행 / 소계 행**
   - 일부 DOCX에 빈 구분행(empty row)이 있음
   - 소계/합계 행도 매칭 대상
   - → is_empty, is_subtotal, is_total 플래그 확인
